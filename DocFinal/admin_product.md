# 管理者端：商品管理模組

---

## 3.1 商品結構

一個商品代表一個主題圖片，底下可以掛多個規格變體。
每個規格變體對應一個已通過審核且完成實體色對應的 production_job 結果。

```
商品（product）
  ├── 規格變體 A → production_job（入門 × 標準 × 30×40cm）
  ├── 規格變體 B → production_job（中級 × 細緻 × 40×60cm）
  └── 規格變體 C → production_job（進階 × 高級 × 60×60cm）
```

**同一個 production_job 可以出現在多個商品**（例如同時收錄在「母親節系列」與「人像精選」）

---

## 3.2 建立商品

**步驟**

1. 填寫商品基本資訊（標題、描述、封面圖）
2. 從「可用變體池」中選擇要加入的 production_job 結果
   - 可用條件：`approved=true` 且實體色對應已全部完成
3. 為每個變體設定價格（參考系統公式，可手動調整）
4. 選擇是否立即上架

**封面圖與商品圖片**

- 預設使用某個變體的 `filled_template.png` 作為封面圖
- 可另外手動上傳封面圖覆蓋
- 支援上傳多張商品圖片（如成品實拍、細節圖），商品詳情頁可左右切換瀏覽
- 圖片存於 Firebase Storage（公開），順序可拖曳排序

---

**系列設定（選填）**

- 新增商品時可選擇「加入現有系列」或「建立新系列」
- 建立新系列：填寫系列名稱（選填描述）→ 立即建立並自動選入
- 選擇現有系列後，設定此商品在系列內的排序（`series_order`）

---

## 3.3 規格變體

每個變體的資訊來自對應的 production_job，顯示給客戶的選項為：

- 畫布尺寸（如 30×40、40×60、60×60 cm）
- 難易度（入門 / 初級 / 中級 / 進階）
- 細緻度（粗糙 / 標準 / 細緻 / 高級）

客戶選擇組合後，系統帶出對應變體的預覽圖與價格。
若某個組合未開放，該選項顯示為不可選。

---

## 3.4 定價

```
售價 = (基礎成本 + 色數 × NT$1) × (1 + 細緻度加成率)
```

| 細緻度 | 加成率 |
|------|------|
| 粗糙 | 60% |
| 標準 | 75% |
| 細緻 | 80% |
| 高級 | 85% |

系統在建立變體時自動帶入公式建議售價（`price_formula_base`），管理員可手動調整最終售價（`price`）。
每個變體各自設定價格，同一商品不同規格可以有不同售價。

> 完整公式說明見 [pricing_formula.md](pricing_formula.md)。

## 3.5 庫存與上架狀態

商品的可售狀態由**兩個層級**決定：

**層級一：管理員控制（`products.status`）**

| 值 | 說明 |
|----|------|
| draft | 草稿，前台不顯示 |
| on_sale | 上架中，前台顯示 |
| off_sale | 已下架，前台不顯示（保留資料） |

**層級二：庫存計算（on_sale 時的派生狀態）**

以**變體**為單位計算（見 admin_color.md §2.8「變體可供件數計算」）：

| 派生狀態 | 條件 |
|---------|------|
| 現貨 | 可供件數 > 0 |
| 預購 | 可供件數 = 0（但 status = on_sale）|

**商品層級的派生狀態：**
- 所有 variants 都有現貨 → 商品顯示「正常販售」
- 部分 variants 預購 → 商品顯示「部分預購」（前台依使用者選擇的規格顯示對應狀態）
- 所有 variants 都預購 → 商品顯示「預購中」

庫存狀態**自動計算**，不存於資料庫。

## 3.6 標籤管理

標籤掛在商品層級，用於主題分類（如「母親節」、「人像」、「風景」）。
一個商品可有多個標籤，一個標籤可對應多個商品（many-to-many）。

**標籤管理操作**

| 操作 | 說明 |
|------|------|
| 新增標籤 | 填寫標籤名稱 |
| 編輯標籤 | 修改標籤名稱 |
| 刪除標籤 | 同時移除所有商品與此標籤的關聯 |
| 查看標籤列表 | 顯示所有標籤及各標籤的商品數量 |

**商品設定標籤**：新增或編輯商品時，可從現有標籤中多選指定。

## 3.7 商品管理操作

| 操作 | 說明 |
|------|------|
| 新增商品 | 見 3.2（含設定標籤）|
| 編輯商品 | 修改標題、描述、封面圖、標籤 |
| 新增 / 移除變體 | 在現有商品中新增或移除規格選項 |
| 調整變體價格 | 單獨修改某個規格的售價 |
| 上架 / 下架 | 控制商品是否在商店顯示。下架時若該商品有任一變體存在未完成訂單（order_items 對應訂單狀態非 completed / cancelled），顯示警告：「此商品有 X 筆進行中訂單，下架後不影響現有訂單，確定繼續？」管理員可強制確認下架，已成立訂單不受影響，僅新客戶無法選購。|
| 查看商品列表 | 分頁、篩選（上架狀態、難易度、畫布尺寸、標籤）|
| 查看銷售統計 | 各變體的訂單數、營收（連結至訂單管理）|

## 3.8 客製服務

當客戶有特殊需求（自帶照片 或 想要目錄圖片的非現有規格），可提交客製申請。
管理員審核後在後台計算報價，系統通知客戶回站確認。

**客製服務兩種類型**

| 類型 | 說明 |
|------|------|
| 客製照片 | 客戶上傳自己的照片（家庭照、寵物照等），管理員製作專屬套件 |
| 客製規格 | 客戶對現有目錄圖片想要不同的尺寸 / 難易度 / 細緻度組合 |

**客戶端申請表單**

| 欄位 | 客製照片 | 客製規格 |
|------|---------|---------|
| 服務類型選擇 | ✓ | ✓ |
| 上傳照片 | ✓（必填）| ✗ |
| 參考商品 | ✗ | ✓（必填，從商品頁點入必帶，獨立入口不提供）|
| 畫布尺寸偏好 | 選填 | 必填 |
| 難易度偏好 | 必填（含「讓管理員建議」）| 必填（含「讓管理員建議」）|
| 細緻度偏好 | **不顯示**（管理員決定）| 必填（含「讓管理員建議」）|
| 備注 / 特殊需求 | 選填 | 選填 |

**完整流程**

```
客戶填寫申請表單 → 提交
    ↓
管理員後台收到申請通知（狀態：quote_pending）
    ↓
管理員審核 → 使用定價公式計算報價 → 填入金額 → 送出報價
    ↓
系統寄 email 給客戶（含規格摘要 + 報價金額 + 確認連結）
連結直接導向報價確認頁面，需登入後查看
    ↓
客戶在報價確認頁選擇：
  ├── 確認 → 進入結帳付款流程（24 小時內付款）
  └── 拒絕 → 申請取消，管理員收到通知
    ↓
付款確認後 → 管理員開始製作 → 正常出貨流程
```

**報價有效期規則**
- 預設有效期：1 天（管理員送出報價起算）
- 客戶可在報價確認頁點選「延長考慮時間」→ 再延 1 天（每筆申請限延一次）
- 延長後系統更新 `quote_expires_at`，並寄 email 通知管理員
- 逾期未回應（含延長後仍未回應）→ 自動取消，狀態改為 `quote_expired`

## 3.10 資料庫欄位

**商品表（products）**

| 欄位 | 說明 |
|------|------|
| id | 主鍵 |
| title | 商品標題 |
| description | 商品描述 |
| cover_image_url | 封面圖（Firebase Storage） |
| series_id | FK → product_series.id（nullable，不屬於系列則為 null）|
| series_order | 系列內排序（nullable）|
| status | draft / on_sale / off_sale |
| created_at | 建立時間 |
| updated_at | 最後更新時間 |

**商品圖片表（product_images）**

| 欄位 | 說明 |
|------|------|
| id | 主鍵 |
| product_id | FK → products.id |
| image_url | Firebase Storage 路徑（公開）|
| sort_order | 排列順序 |
| created_at | 建立時間 |

**商品規格變體表（product_variants）**

| 欄位 | 說明 |
|------|------|
| id | 主鍵 |
| product_id | FK → products.id |
| production_job_id | FK → production_jobs.id |
| price | 售價（手動設定，參考公式建議） |
| price_formula_base | 公式計算的建議定價（記錄用） |
| is_active | 是否開放選購 |
| created_at | 建立時間 |

> 備註：difficulty、detail、canvas_w_cm、canvas_h_cm 等資訊從 production_job 讀取，不重複儲存。
> 同一個 production_job_id 可出現在多筆 product_variants（跨不同商品）。
> 變體庫存狀態由 production_job 對應的 palette_color_mappings → physical_colors.stock 自動計算，不另存欄位。

**標籤表（tags）**

| 欄位 | 說明 |
|------|------|
| id | 主鍵 |
| name | 標籤名稱（唯一）|
| created_at | 建立時間 |

**商品標籤關聯表（product_tags）**

| 欄位 | 說明 |
|------|------|
| product_id | FK → products.id |
| tag_id | FK → tags.id |

> 主鍵為 (product_id, tag_id) 複合主鍵。刪除標籤時 CASCADE 移除所有關聯。
