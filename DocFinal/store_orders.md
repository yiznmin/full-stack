# 用戶商店端：訂單管理

---

## 訂單編號格式

```
PL-YYYYMMDD-XXXXXX
例：PL-20260417-000001
```

> 流水號全域遞增，不依日期重置。6 位數上限 999999 筆；超過時自動擴展為 7 位或更多（前端 parser 不 hardcode 位數）。

---

## 訂單列表

分三個頁籤顯示，訂單依狀態自動歸類：

| 頁籤 | 包含狀態 |
|------|---------|
| 未付款 | pending_payment |
| 出貨中 | paid / processing / shipped |
| 已完成 | completed / cancelled / refund_processing / refunded / partially_refunded |

每筆訂單顯示：訂單編號、建立日期、商品縮圖（最多 3 個）、總金額、目前狀態

**付款核對表單入口**

「未付款」頁籤中，每筆訂單旁顯示「填寫付款資訊」連結：
- 點選後展開表單：轉帳金額、轉帳日期、轉帳時間、帳號末五碼、備注
- 提交後管理員後台可對照核實
- 管理員確認付款後，訂單自動移至「出貨中」

## 訂單詳情

| 區塊 | 說明 |
|------|------|
| 訂單資訊 | 訂單編號、建立時間、付款期限（pending_payment 時顯示倒數）|
| 商品明細 | 商品名稱、規格快照、單價、數量（現貨 + 預購分開顯示）|
| 金額明細 | 小計、折扣、運費、總計 |
| 收件資訊 | 取貨方式、收件人、地址或門市 |
| 付款資訊 | 付款帳號（pending_payment 時顯示）、付款確認時間 |
| 出貨資訊 | 物流追蹤號、取貨說明（shipped 後顯示）|
| 訂單狀態 | 目前狀態標示 |
| 取消按鈕 | 僅 pending_payment 狀態顯示，paid 後隱藏 |
| 確認收貨按鈕 | 僅 `order.status = shipped`（所有 shipments 均已出貨）後顯示，部分出貨期間不顯示 |
| 付款後狀態訊息 | paid 後顯示「我們將盡快為您製作與出貨」|
| 退款確認區塊 | `orders.status ∈ {refunded, partially_refunded}` 時顯示,分兩種狀態(見下)|

**退款確認區塊**

管理員標記退款完成後,客戶於訂單詳情頁會看到:

| 條件 | 顯示內容 |
|------|---------|
| `refund_confirmed_at IS NULL` | 紅色提示「已退款 NT$X,請確認已收到退款」+「確認已收到退款」按鈕 |
| `refund_confirmed_at IS NOT NULL` | 綠色標示「退款已完成,您已於 YYYY-MM-DD 確認收到」|

**客戶確認流程**
- 點「確認已收到退款」按鈕 → 後端 UPDATE `orders.refund_confirmed_at = now()`
- 也可從退款完成 email 內的連結直接確認(連結格式如 `/orders/:id?action=confirm_refund`,點擊自動觸發)
- 確認後頁面即時更新為綠色「退款已完成」

**取消確認流程**

```
點「取消訂單」
    ↓
跳出確認視窗：「確定取消訂單 #PL-xxx？取消後無法恢復」
    ↓ 確認
狀態改為 cancelled，顯示「訂單已取消」
```

**預購項目出貨偏好**

結帳時若購物車含有預購數量，顯示選項：

| 選項 | 說明 |
|------|------|
| 合併出貨 | 等所有預購到貨後統一出貨，運費一次 |
| 分開出貨 | 現貨先出，預購到貨後再出，運費仍只收一次 |

> `orders.shipping_preference` 欄位儲存選擇（`together` / `separate`）

## 客製申請列表

會員中心獨立頁面，顯示所有 `custom_requests` 記錄：

- custom_photo（客製照片）
- custom_spec（客製規格，從商品頁提交）

每筆顯示：申請類型、提交時間、目前狀態、操作

| 狀態 | 顯示文字 | 可操作 |
|------|---------|--------|
| quote_pending | 等待報價中 | **自由修改申請內容**(照片/偏好/備註)|
| negotiating | 洽談中(製作中)| — (鎖定,不可改申請內容)|
| quote_sent | 報價已送出,請確認 | 點「待確認」→ 進入報價頁(顯示初稿預覽 + 報價 + 確認/要求修改/拒絕按鈕,等價 `/custom/quote/:token`)|
| draft_revision | 修改中(已要求修改初稿)| — |
| quote_confirmed | 已確認,等待付款 | 前往付款 |
| quote_rejected | 已拒絕 | 重新申請 |
| quote_expired | 已過期 | — |

**申請詳情頁內容(會員中心 `/custom/requests/:id`)**

客戶於會員中心點進客製申請詳情可看到:

| 區塊 | 顯示內容 |
|------|---------|
| 申請基本資訊 | 類型(custom_photo / custom_spec)、提交時間、目前狀態 |
| 提交內容 | 照片(custom_photo)、參考商品(custom_spec)、尺寸偏好、難易度、備註 |
| 目前狀態與可操作 | 依狀態決定(見上表)|
| 訊息對談區 | 所有 custom_request_messages 往來記錄 |

> **quote_pending 自由修改的實作**:更新照片時同 E18 機制(舊照片立即 Firebase 刪除、覆蓋 photo_url);更新尺寸偏好/難易度/備註等欄位直接 UPDATE。
> **negotiating 鎖定**:後端驗證,若 status 不為 quote_pending 拒絕修改請求,回 409。
> **quote_sent 點「待確認」**:進入頁面顯示初稿預覽圖 + 報價 + 三個操作鈕(確認/要求修改/拒絕),內容等同 email 連結 `/custom/quote/:token`,僅入口差異(從會員中心進入已登入,不需驗證 token)。

**訊息系統**

客製申請詳情頁內建訊息對談區：
- 申請送出時系統自動發歡迎訊息
- 客戶可在此頁回覆管理員訊息
- 用戶在對話頁時：新訊息透過 SSE 即時推送，不發 email
- 用戶離開頁面時：新訊息發 email 通知，附連結導回對話頁

## 生產進度追蹤

每個訂單項目獨立顯示生產進度：

| 狀態(後端)| 客戶顯示文字 |
|------|------------|
| pending | 等待備貨 |
| in_production | 製作中 |
| manufacturing | **備貨中** |
| packaging | **備貨中** |
| ready_to_ship | 備貨完成 |
| shipped | 已出貨 |

> 後端狀態分 `manufacturing`(印製模板)與 `packaging`(打包),但**客戶端統一顯示「備貨中」**,避免技術細節干擾客戶理解。管理員後台仍顯示兩個獨立狀態以利操作區分。

- 狀態更新時系統自動發 email 通知
- 管理員填寫的 `notes` 欄位客戶可見

## 付款說明頁

訂單建立後顯示：
- 付款帳號（銀行帳號）
- 付款金額
- 付款期限（24 小時倒數）
- 注意事項（轉帳後可填寫付款核對表單，管理員確認付款後更新訂單狀態）
- **付款核對表單**：填寫轉帳金額、日期、時間、帳號末五碼，方便管理員比對

## 報價確認頁（客製服務）

客戶收到報價 email 後，點連結進入此頁：

- **不需登入查看**（token 本身為一次性短效憑證，24h 或延長後的 48h 內有效）
- Token 過期後訪問此連結 → 導向登入頁，登入後導回 `/custom/requests/:id` 查看申請
- 若客戶已登入狀態下訪問此連結 → 額外驗證該 token 對應的 custom_request.user_id 是否等於當前登入用戶（防止誤點他人信件）
- 顯示：規格摘要、報價金額、報價有效期限（倒數）
- 操作：
  - **確認報價** → 進入客製結帳步驟（見下方）
  - **拒絕報價** → 申請取消，填寫原因（選填）
  - **延長考慮時間** → 再延 1 天（每筆限一次，延長後按鈕消失）

**客製結帳步驟（確認報價後）**

```
1. 填寫 / 選擇收件資訊
   （從已儲存的 shipping_profiles 選，或新填）
   - 超商取件：透過第三方選店 API 選擇門市
    ↓
2. 確認金額
   - 商品：報價金額（quoted_price）
   - 運費：依取貨方式（宅配 NT$120 / 超商 NT$70，不適用免運條件）
   - 折扣：不支援折扣券與 auto_checkout 促銷（客製服務為一對一報價，不套用任何折扣）
   - 總計 = quoted_price + 運費
    ↓
3. 送出 → 系統建立訂單（status: pending_payment）
   → 顯示付款帳號 + 24 小時付款期限
   → 同時發送訂單確認 email
```

> 客製申請訂單與購物車訂單各自獨立，不合併結帳，運費各自計算。

## 訂單完成判定

訂單標記為 `completed` 採三軌機制：

| 方式 | 說明 |
|------|------|
| 物流自動更新 | ECpay Webhook 收到「已取貨/已投遞」→ 系統自動改為 completed |
| 客戶主動確認 | 訂單詳情頁顯示「確認已收到商品」按鈕，點選後立即 completed |
| 逾期提醒 | 出貨後 14 天無取貨確認 → 系統通知管理員確認，由管理員手動處理 |

> 完成後才觸發回饋券（`spend_reward` / `returning_loyal`）的發放。
